Lección 0 — Prerrequisitos & Verificar estado inicial

Instala Git y asegúrate acceso al repo. Comandos para validar el estado inicial después de tu primer git push a main:

# desde la carpeta del repo
git remote -v
git branch -a
git status

Salida esperada (ficticia):

origin https://github.com/softwareincorp0223/frontend-ERP.git (fetch) origin https://github.com/softwareincorp0223/frontend-ERP.git (push) * main remotes/origin/main On branch main nothing to commit, working tree clean

Lección 1 — Crear la rama develop y subirla

Comandos y salidas esperadas para crear develop a partir de main:

git checkout -b develop

git push -u origin develop

git branch -a

Salida esperada (ficticia):

Switched to a new branch 'develop' To https://github.com/softwareincorp0223/frontend-ERP.git * [new branch] develop -> develop Branch 'develop' set up to track remote branch 'develop' from 'origin'. develop * main remotes/origin/develop remotes/origin/main

Comandos para el colaborador:

Lección 3 — Crear feature/login, editar, commitear y pushear

Ejemplo práctico: vamos a crear src/login.js con una función mock. Sigue estos pasos:

# partir de develop
git checkout develop
git pull origin develop

# crear rama de feature
git checkout -b feature/login

# crear archivo y commitear
mkdir -p src
cat > src/login.js <<'EOF'
export function mockLogin(user, pass) {
  return user === 'admin' && pass === 'secret';
}
EOF

git add src/login.js
git commit -m "feat(login): add mock login util for local testing"

git push -u origin feature/login

Salida esperada (ficticia):

On branch feature/login Untracked files: src/ [feature/login 1a2b3c4] feat(login): add mock login util for local testing 1 file changed, 7 insertions(+) create mode 100644 src/login.js To https://github.com/softwareincorp0223/frontend-ERP.git * [new branch] feature/login -> feature/login Branch 'feature/login' set up to track remote branch 'feature/login' from 'origin'.

Lección 4 — Crear Pull Request (feature → develop)

Opción A: desde la interfaz web de GitHub (recomendado). Opción B: desde la terminal con gh CLI.

gh pr create --base develop --head feature/login --title "feat(login): add mock login util" --body "Agregar util de login falso para pruebas unitarias"

Salida esperada (ficticia):

Created pull request #3: feat(login): add mock login util https://github.com/softwareincorp0223/frontend-ERP/pull/3

Lección 5 — Revisar y mergear PR a develop

Dos formas: fusionar desde GitHub o hacerlo localmente. Ejemplo de merge local:

git checkout develop
git pull origin develop

git fetch origin feature/login:feature/login

git merge --no-ff feature/login -m "chore: merge feature/login into develop"

git push origin develop

Salida esperada (ficticia):

Merge made by the 'recursive' strategy. src/login.js | 7 +++++++ 1 file changed, 7 insertions(+)

Lección 6 — Mantener ramas actualizadas (merge vs rebase)

Actualizar tu rama feature con los últimos cambios de develop.

Merge (sencillo): 
git checkout feature/login
git fetch origin
git merge origin/develop
git push origin feature/login
Rebase (historial más lineal): 
git checkout feature/login
git fetch origin
git rebase origin/develop
# resolver conflictos si aparecen
git push --force-with-lease origin feature/login

Consejo: si no están acostumbrados a reescribir historia, privilegien merge.

Lección 7 — Release: merge develop → main y tag

Pasos cuando develop está probado:

git checkout main
git pull origin main

git checkout develop
git pull origin develop

# merge a main
git checkout main
git merge --no-ff develop -m "release: merge develop into main (v0.1.0)"

# tag
git tag -a v0.1.0 -m "v0.1.0 - primera release"
 
git push origin main
git push origin v0.1.0

Salida esperada (ficticia):

Merge made by the 'recursive' strategy. ... files changed ... * [new tag] v0.1.0 -> v0.1.0

Lección 8 — Limpieza: borrar ramas feature

git push origin --delete feature/login
git branch -d feature/login
Deleted branch feature/login (was abc1234). Deleted remote branch feature/login (was abc1234).

Lección 9 — Hotfix para producción

Flujo para corrección urgente desde main:

git checkout main
git pull origin main

git checkout -b hotfix/fix-login
# aplicar fix
git add .
git commit -m "fix(login): fix faulty mock login edge case"

git push -u origin hotfix/fix-login
# abrir PR a main o merge directo y luego:

git checkout develop
git pull origin develop
git merge main
git push origin develop
(Después del merge a main) develop se actualiza con el fix para no perderlo.

Lección 10 — Ejemplo didáctico de conflicto y resolución

Reproducimos un conflicto en README.md y lo resolvemos:

# desde develop
# crear feature/a
git checkout -b feature/a
echo "Feature A line" >> README.md
git add README.md
git commit -m "feat: add line from feature A"
git push -u origin feature/a

# crear feature/b
git checkout develop
git checkout -b feature/b
echo "Feature B line" >> README.md
git add README.md
git commit -m "feat: add line from feature B"
git push -u origin feature/b

# merge feature/a a develop
git checkout develop
git merge --no-ff feature/a
git push origin develop

# intentar merge feature/b a develop -> conflicto
git merge feature/b

Salida esperada de conflicto:

Auto-merging README.md CONFLICT (content): Merge conflict in README.md Automatic merge failed; fix conflicts and then commit the result.

Resolver editando el archivo (quitar marcadores <<<<<< ... >>>>>) y luego:

git add README.md
git commit -m "fix: resolve conflict between feature/a and feature/b"
git push origin develop

Resumen y Buenas Prácticas

  • Ramas: main (producción), develop (integración), feature/*, hotfix/*.
  • Commits: mensajes claros, considera conventional commits.
  • Protege main en GitHub (no push directo, requierir PRs y checks).
  • Usa CI para tests/lint antes de mergear.
  • Asigna revisores y mantiene PRs pequeños.
